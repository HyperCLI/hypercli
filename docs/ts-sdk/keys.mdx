## Overview

Use `client.keys` to manage account API keys from TypeScript apps.

The public `/api/keys` surface is Orchestra-authoritative. HyperClaw proxies and caches key state, but the issued keys themselves live in Orchestra.

```ts
import { HyperCLI } from "@hypercli.com/sdk";

const client = new HyperCLI({ apiKey: process.env.HYPER_API_KEY });
```

The TypeScript SDK exposes:

- `client.keys.create(...)`
- `client.keys.list()`
- `client.keys.get(...)`
- `client.keys.rename(...)`
- `client.keys.disable(...)`

## Create Scoped Keys

Tags define access.

Common baseline tags:

- `*:*`
- `jobs:*`
- `jobs:self`
- `renders:*`
- `flows:*`
- `renders:self`
- `files:*`
- `files:self`
- `agents:*`
- `agents:self`
- `models:*`
- `voice:*`
- `user:*`
- `user:self`
- `api:*`
- `api:self`

Common selector tags:

- `team=dev`
- `customer=acme`
- `project=alpha`
- `agent:<agent-uuid>`
- `runtime=agent`

Example:

```ts
const scoped = await client.keys.create("team-dev-app", [
  "jobs:self",
  "renders:self",
  "user:self",
  "team=dev",
]);

console.log(scoped.apiKey);
```

That key can be handed to your own backend, frontend proxy, or worker app.

New keys are deny-by-default unless tags are supplied. Use `*:*` only for trusted full-access keys or legacy migration.

For HyperClaw inference and voice:

- `models:*` is enough for `GET /v1/models`
- `voice:*` is the public voice family contract for `/agents/voice/*`
- new keys are still deny-by-default overall, so only the tags you attach are allowed

## Integration Pattern: Tag-Scoped Key + Tagged Agent

This is the primary downstream integration pattern:

1. create an API key with a selector tag
2. create an agent carrying the same tag
3. start and stop that agent with the generated key

```ts
import { HyperCLI } from "@hypercli.com/sdk";

const root = new HyperCLI({
  apiKey: process.env.HYPER_API_KEY!,
  apiUrl: "https://api.hypercli.com",
  agentsApiBaseUrl: "https://api.hypercli.com/agents",
});

// 1. Mint a narrowed key with one selector tag plus the agents self-family baseline.
const scoped = await root.keys.create("customer-user123", [
  "agents:self",
  "customer=user123",
]);

// 2. Create an agent carrying the same selector tag.
const agent = await root.deployments.create({
  name: "customer-user123-agent",
  size: "small",
  start: false,
  tags: ["customer=user123"],
});

console.log("created agent", agent.id);
console.log("scoped key", scoped.apiKey);

if (!scoped.apiKey) {
  throw new Error("Expected full API key on create");
}

// 3. Use the generated key directly against the deployments API.
const scopedClient = new HyperCLI({
  apiKey: scoped.apiKey,
  apiUrl: "https://api.hypercli.com",
  agentsApiBaseUrl: "https://api.hypercli.com/agents",
});

await scopedClient.deployments.start(agent.id);
console.log("started", agent.id);

await scopedClient.deployments.stop(agent.id);
console.log("stopped", agent.id);
```

Why this is the integration pattern:

- the key carries `customer=user123`
- the agent carries `customer=user123`
- `agents:self` gives the key the agents-family baseline
- the backend then allows the key to act only on resources carrying the same selector tag

Use this when your platform needs to issue one narrowed key per downstream scope, such as:

- `customer=user123`
- `project=alpha`
- `tenant=acme`

Use `=` for your own selectors. `:` is reserved for built-in system scopes.

## Child Keys For One Exact Agent

For embedded agent clients, mint an exact-agent child key:

```ts
const agent = await client.deployments.get("agent-id");
const child = await client.deployments.createScopedKey(agent.id, "agent-client");

console.log(child.api_key);
```

The reserved exact-agent selector is `agent:<agent-uuid>`. `:` stays reserved for built-in scopes; use `=` only for your own metadata tags. Do not use `agents:<uuid>` or `agent=<name>`.

That child key is intended for:

- viewing that agent
- start/stop
- matched logs/state
- inference access through the dedicated inference token endpoint

Exact-agent flow example:

```ts
import { HyperCLI } from "@hypercli.com/sdk";

const root = new HyperCLI({
  apiKey: process.env.HYPER_API_KEY!,
  apiUrl: "https://api.hypercli.com",
  agentsApiBaseUrl: "https://api.hypercli.com/agents",
});

const agent = await root.deployments.create({
  name: "single-agent-client",
  size: "small",
  start: false,
});

const child = await root.deployments.createScopedKey(agent.id, "single-agent-client");

if (!child.api_key) {
  throw new Error("Expected full child key on create");
}

const exactClient = new HyperCLI({
  apiKey: child.api_key,
  apiUrl: "https://api.hypercli.com",
  agentsApiBaseUrl: "https://api.hypercli.com/agents",
});

await exactClient.deployments.start(agent.id);
await exactClient.deployments.stop(agent.id);
```

Use this exact-agent form only when one key should control one exact agent. For normal downstream integrations, prefer the selector-tag flow above.

## List, Inspect, Rename, Disable

```ts
const keys = await client.keys.list();
const key = await client.keys.get(keys[0]!.keyId);
const renamed = await client.keys.rename(key.keyId, "frontend-prod");
await client.keys.disable(renamed.keyId);
```

`list()` and `get()` return masked keys. The full `apiKey` value is only returned on creation.

## App Pattern

Typical backend app flow:

```ts
const root = new HyperCLI({ apiKey: process.env.HYPER_API_KEY });

const customerKey = await root.keys.create("customer-app", [
  "jobs:self",
  "user:self",
  "team=customer-a",
]);
```

Typical agent app flow:

```ts
const root = new HyperCLI({
  apiKey: process.env.HYPER_API_KEY,
  agentApiKey: process.env.HYPER_AGENTS_API_KEY,
});

const child = await root.deployments.createScopedKey("agent-id", "embedded-agent-client");
```
