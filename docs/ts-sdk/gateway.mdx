---
title: Gateway Client
description: Low-level TypeScript gateway client for OpenClaw pairing, config, chat, files, channels, sessions, and cron.
---

`GatewayClient` is the low-level WebSocket client for the OpenClaw protocol.

Use it when you want direct access to:

- config RPC
- raw events
- chat/session control
- workspace files
- paired node command RPC
- channel login/status flows
- cron RPC

## Create

```ts
import { GatewayClient } from "@hypercli.com/sdk";

const gateway = new GatewayClient({
  url: "wss://my-agent.hypercli.app",
  gatewayToken: "...",
  deploymentId: "agent-id",
  apiKey: "hyper_api_...",
  apiBase: "https://api.hypercli.com/agents",
  autoApprovePairing: true,
});
```

Important options:

- `url`
- `gatewayToken`
- `deploymentId`
- `apiKey`
- `apiBase`
- `autoApprovePairing`
- `timeoutMs`
- `chatTimeoutMs`
- `clientId`
- `clientMode`
- `role`
- `scopes`
- `caps`

Default handshake behavior for the TypeScript SDK:

- `clientId="cli"`
- `clientMode="cli"`
- no `Origin` header unless you pass `origin` explicitly

That matters because OpenClaw applies browser/control-ui origin checks to
`webchat`/control-ui clients and to any connection carrying an `Origin` header.

## Auth Model

This client does not use browser `/_jwt_auth` semantics.

It connects to the root agent host and authenticates at the OpenClaw protocol layer using:

- gateway token
- device identity
- pairing / cached device token

## Readiness

```ts
await gateway.waitReady(300_000);
```

This is the recommended way to wait for a fresh agent after the pod reaches `RUNNING`.

## Chat And Sessions

Two patterns are useful:

### Raw RPC + event listeners

```ts
gateway.onEvent((event) => console.log(event));
await gateway.sendChat({ message: "Summarize this repository", sessionKey: "docs" });
```

### Higher-level history/session helpers

- `chatHistory()`
- `chatAbort()`
- `sessionsList()`
- `sessionsPreview()`
- `sessionsReset()`

## Nodes

Nodes are separate machines or processes connected to the same gateway with
`role: "node"`. A node declares a command surface; an operator approves that
surface and invokes commands through the gateway.

Operator-side methods:

- `nodesList()`
- `nodeDescribe(nodeId)`
- `nodePairList()`
- `nodePairApprove(requestId)`
- `nodePairReject(requestId)`
- `nodePairRemove(nodeId)`
- `nodeRename(nodeId, displayName)`
- `nodeInvoke(nodeId, command, params?, timeoutMs?)`

Node-side helper:

```ts
import { NodeServer } from "@hypercli.com/sdk";

const node = new NodeServer(
  {
    "shell.run": async () => ({ stdout: "ok", exitCode: 0 }),
  },
  {
    url: "wss://my-agent.hypercli.app",
    gatewayToken: "...",
    nodeId: "node-abc",
  },
);

await node.start();
```

`NodeServer` connects with `role: "node"`, sends `commands` during the gateway
handshake, listens for `node.invoke.request`, and replies with
`node.invoke.result`. Success replies contain `payloadJSON`; handler failures
return an error object.

Pairing has two layers:

- device pairing, approved with `device.pair.approve`
- node command-surface pairing, approved with `nodePairApprove(requestId)`

For custom commands such as `shell.run`, `fs.read`, or `fs.write`, the gateway
policy must also allow the command names through `gateway.nodes.allowCommands`.
If you patch that config in a containerized agent, restart the whole agent and
reconnect clients before pairing or invoking nodes. A config patch alone can
leave the running gateway on the old command policy.

## Recommendation

Most application code should not instantiate `GatewayClient` directly. Prefer:

- `OpenClawAgent.gateway()`
- `OpenClawAgent.connect()`
