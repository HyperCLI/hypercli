
## What The Python SDK Covers

The Python SDK has two distinct halves:

- HyperCLI platform APIs
  - billing
  - jobs
  - instances
  - renders
  - files
  - keys
  - user
- HyperClaw / agent APIs
  - `client.agent` for plan and model metadata
  - `client.deployments` for agent lifecycle and runtime access

```python
from hypercli import HyperCLI

client = HyperCLI(
    api_key="hyper_...",      # main HyperCLI API key
    agent_api_key="sk-...",   # HyperClaw agent key
)
```

## Authentication Model

The constructor resolves credentials in this order:

1. explicit constructor args
2. environment variables
3. `~/.hypercli/config`

Common environment variables:

- `HYPERCLI_API_KEY`
- `HYPERCLI_API_URL`
- `AGENTS_API_BASE_URL`
- `AGENTS_WS_URL`
- `HYPERCLAW_API_KEY`
- `HYPERCLAW_API_BASE`

Use `agent_api_key` when your HyperClaw key is different from your main HyperCLI key.

## Main Namespaces

### Platform

- `client.billing`
- `client.jobs`
- `client.instances`
- `client.renders`
- `client.files`
- `client.keys`
- `client.user`

### HyperClaw

- `client.agent`
  - plans
  - model catalog
  - convenience provider config helpers
- `client.deployments`
  - create / start / stop / delete
  - exec
  - file copy and file APIs
  - log and shell WebSocket helpers
  - OpenClaw gateway access

## Quick Start

```python
from hypercli import HyperCLI

client = HyperCLI(api_key="hyper_...", agent_api_key="sk-...")

agent = client.deployments.create_openclaw(
    name="docs-demo",
    start=True,
    env={"APP_ENV": "prod"},
)

agent = agent.wait_running(timeout=300)
print(agent.public_url)   # https://docs-demo.hyperclaw.app
print(agent.desktop_url)  # https://desktop-docs-demo.hyperclaw.app
```

## Current Agent Route Model

For OpenClaw-backed agents, the platform defaults are:

- root host = gateway / agent
- `desktop-<host>` = desktop
- extra routes come from the `routes` dict you pass at create/start time

That means:

- `Agent.public_url` is the root host
- `Agent.desktop_url` and `Agent.vnc_url` point to `desktop-<host>`
- `Agent.shell_url` is only populated if you explicitly create a `shell` route

There is no default `shell-<host>` platform route anymore.

## Recommended Access Patterns

### Use backend `/ws/*` for logs and shells

Preferred helpers:

- `agent.logs_stream_ws()`
- `agent.shell_connect()`
- `client.deployments.logs_stream_ws(...)`
- `client.deployments.shell_connect(...)`

These use backend-issued short-lived tokens and the backend `/ws/logs/{agent_id}` and `/ws/shell/{agent_id}` endpoints.

### Use `OpenClawAgent.gateway()` for gateway work

Preferred helpers:

- `agent.gateway()`
- `agent.connect()`
- `agent.wait_ready()`
- `agent.config_get()`
- `agent.chat_send()`

These connect to the root agent host, not `openclaw-<host>`.

### Treat direct executor helpers as legacy

The SDK still contains some direct executor-era helpers:

- `health()`
- `chat_stream()`
- `logs_stream()`

They only make sense if you have explicitly created an executor-style route. For current OpenClaw agents, prefer backend `/ws/*` and gateway RPC instead.
