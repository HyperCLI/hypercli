`GatewayClient` is the low-level async WebSocket client for the OpenClaw gateway protocol.

Use it when you want direct access to:

- config RPC
- chat/session RPC
- workspace file RPC
- paired node command RPC
- channel auth flows
- cron management
- raw event handling

In most apps, you will construct it via `OpenClawAgent.gateway()`.

## Connection Model

Gateway auth is application-level.

Current remote flow:

- connect to `wss://<agent-host>`
- present gateway token
- complete device challenge
- auto-approve pairing if the SDK is bound to a deployment and API key
- reuse cached device auth when possible

This is not the same as browser `/_jwt_auth` route bootstrap.

## Create A Client

```python
from hypercli.gateway import GatewayClient

gw = GatewayClient(
    url="wss://my-agent.hypercli.app",
    gateway_token="...",
    deployment_id="agent-id",
    api_key="hyper_api_...",
    api_base="https://api.hypercli.com/agents",
    auto_approve_pairing=True,
)
```

Common options:

- `url`
- `gateway_token`
- `deployment_id`
- `api_key`
- `api_base`
- `auto_approve_pairing`
- `timeout`
- `chat_timeout`
- `client_id`
- `client_mode`
- `role`
- `scopes`
- `caps`

Default handshake behavior for the Python SDK:

- `client_id="cli"`
- `client_mode="cli"`
- no `Origin` header unless you pass `origin=...` explicitly

That matters because OpenClaw applies browser/control-ui origin checks to
`webchat`/control-ui clients and to any connection carrying an `Origin` header.

## Lifecycle

```python
async with gw:
    print(gw.version, gw.protocol)
```

Or manually:

```python
await gw.connect()
...
await gw.close()
```

Properties:

- `version`
- `protocol`
- `is_connected`
- `pending_pairing`

## Events

Register a raw event callback:

```python
unsubscribe = gw.on_event(lambda event: print(event))
```

The callback sees the raw gateway event stream, which is useful for:

- custom chat UX
- progress events
- approval flows
- debugging provider behavior

## Readiness

Use readiness explicitly after a pod reaches `RUNNING`:

```python
await gw.wait_ready(timeout=300)
```

By default this polls `config.get`, which is a better signal than pod state alone.

## Configuration

```python
config = await gw.config_get()
schema = await gw.config_schema()
await gw.config_patch({"models": {"providers": {"openai": {"apiKey": "hyper_api_..."}}}})
await gw.config_apply(config)
```

Config methods:

- `config_get()`
- `config_schema()`
- `config_patch()`
- `config_set()`
- `config_apply()`

## Chat And Sessions

High-level streaming helper:

```python
async for event in gw.chat_send("Summarize this repo", session_key="docs"):
    print(event)
```

Available helpers:

- `chat_send()`
- `send_chat()`
- `chat_history()`
- `chat_abort()`
- `sessions_list()`
- `sessions_preview()`
- `sessions_reset()`

The Python client now handles:

- modern `chat.content` / `chat.done`
- legacy `event == "chat"` final states
- run correlation using both client idempotency and server `runId`

## Nodes

Nodes are separate machines or processes connected to the same gateway with
`role="node"`. A node declares a command surface; an operator approves that
surface and invokes commands through the gateway.

Operator-side methods:

- `nodes_list()`
- `node_describe(node_id)`
- `node_pair_list()`
- `node_pair_approve(request_id)`
- `node_pair_reject(request_id)`
- `node_pair_remove(node_id)`
- `node_rename(node_id, display_name)`
- `node_invoke(node_id, command, params=None, timeout_ms=None)`

Node-side helper:

```python
from hypercli.gateway import NodeServer

node = NodeServer(
    "wss://my-agent.hypercli.app",
    "node-abc",
    {
        "shell.run": lambda params: {"stdout": "ok", "exitCode": 0},
    },
    gateway_token="...",
)

await node.connect()
```

`NodeServer` connects with `role="node"`, sends `commands` during the gateway
handshake, listens for `node.invoke.request`, and replies with
`node.invoke.result`. Success replies contain `payloadJSON`; handler failures
return an error object.

Pairing has two layers:

- device pairing, approved with `device.pair.approve`
- node command-surface pairing, approved with `node_pair_approve(request_id)`

For custom commands such as `shell.run`, `fs.read`, or `fs.write`, the gateway
policy must also allow the command names through `gateway.nodes.allowCommands`.
If you patch that config in a containerized agent, restart the whole agent and
reconnect clients before pairing or invoking nodes. A config patch alone can
leave the running gateway on the old command policy.

## Recommendation

Use `OpenClawAgent.gateway()` unless you are:

- building a custom transport wrapper
- debugging raw gateway behavior
- implementing a custom pairing or event model
